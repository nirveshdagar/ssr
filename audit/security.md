# Security audit — 2026-04-30

**Verdict**: 13 findings — 0 P0, 6 P1, 5 P2, 2 P3. Auth gating is centralized and tight, secrets are mostly encrypted, SQL is parameterized — but `cf_keys.api_key` is plaintext at rest, `GET /api/settings` returns every secret unredacted to any authenticated session, missing CSRF defense leaves every state-changing route exploitable via cross-site form submit, and several SSH paths interpolate user-controlled data. Not ready for prod without the P1 list closed.

## P0 — Production blockers

(none — nothing meets the P0 bar of "trivially exploitable in prod by an unauthenticated attacker")

## P1 — Must fix before prod

### [P1] CF Global API keys stored plaintext in `cf_keys.api_key`
**File**: lib/repos/cf-keys.ts:30, lib/secrets-vault.ts:24-36
**Issue**: `secrets-vault` only encrypts the `settings` table (via `isSensitive(key)` against a whitelist of setting *keys*). The `cf_keys.api_key` column is stored as raw plaintext — confirmed by `substr(k.api_key, 1, 6) || '...' || substr(k.api_key, length(k.api_key) - 3) AS key_preview` in `listCfKeysWithPreview`. Same row's `api_key` is read in cleartext throughout `cf-key-pool.ts` and posted to CF as `X-Auth-Key`. Note: `cf_workers_ai_keys.api_token` IS encrypted (lib/repos/cf-ai-keys.ts) — only the *DNS* CF Global Keys table is unprotected.
**Why it matters**: Anyone with read access to `data/ssr.db` (backup theft, host compromise, accidental dropbox sync, an `sqlite3 ssr.db` from a co-tenant) walks away with every CF Global Key. CF Global Keys grant full account control — DNS hijack, zone deletion, billing visibility, member manipulation. The entire fleet is downstream of these keys. The secrets-vault module exists specifically to prevent this; it's just not wired up to this column.
**Suggested fix**: Encrypt `cf_keys.api_key` with the existing Fernet vault (mirroring what `cf-ai-keys.ts` already does — `addCfKey` calls `encrypt(apiKey)` before insert; every read decrypts). Add a one-shot boot migration that re-encrypts existing rows (same shape as `encryptExistingAiTokens`). Drop the SQL-level `key_preview` and compute it post-decrypt in JS.

### [P1] `GET /api/settings` returns every secret in plaintext to the session
**File**: app/api/settings/route.ts:63-83
**Issue**: The GET handler iterates `STRING_FIELDS` and returns `getSetting(k)` for every entry — `spaceship_api_secret`, `do_api_token`, `do_api_token_backup`, `serveravatar_api_key`, `sa_dashboard_password`, `smtp_password`, `telegram_bot_token`, `whatsapp_apikey`, `greenapi_api_token`, `twilio_auth_token`, `server_root_password`, `cloudflare_workers_ai_token`, every `llm_api_key_*`. Only `dashboard_password` is special-cased to a boolean.
**Why it matters**: One stolen session cookie (XSS, cookie disclosure, session fixation, malicious browser extension, auth-bypass bug elsewhere) → full credential exfiltration in a single GET. The encrypted-at-rest design is bypassed because the API serves plaintext to any authed caller. Any future XSS in the dashboard is upgraded to "all infra credentials compromised" instead of "session takeover."
**Suggested fix**: Apply the same pattern used for `dashboard_password`: return `*_set` booleans (or last-4-char masks) instead of plaintext for any key in `SECRET_KEYS_EXACT` / matching `SECRET_KEYS_PREFIX`. The Settings UI only needs to know "is this populated" plus the ability to overwrite — it doesn't need to display the existing value. If displayability is a hard UX requirement, add a separate `POST /api/settings/reveal-secret` that requires re-entry of the dashboard password and rate-limits to N reveals/min.

### [P1] No CSRF defense — every state-changing route is cross-site exploitable
**File**: lib/auth-config.ts:40, middleware.ts (entire file)
**Issue**: Session cookie is `sameSite: "lax"`, no CSRF token issued anywhere, no `Origin`/`Referer` check in middleware or any route. `lax` blocks third-party POSTs from cross-site `<form>` submissions in modern browsers, BUT (a) `lax` permits top-level GET navigations carrying the cookie, and several state-changing operations are reachable via plain links / GETs through the audit / read paths (e.g. `GET /api/sa/index-file?domain=…&server_ip=…` reads `index.php` over SSH), and (b) `lax` does NOT block POSTs that come from user-initiated top-level navigations in some browsers, and (c) any subdomain of the deploy domain that gets compromised is treated as same-site.
**Why it matters**: If an authenticated operator visits an attacker-controlled page (or a phishing-domain on a CF zone the attacker also controls), the attacker can fire `POST /api/servers/destroy-all` (with `confirm_phrase=DESTROY ALL`), `POST /api/domains/[domain]/full-delete`, `POST /api/settings` rewriting the dashboard password / DO token / SA password, or `POST /api/sa/upload-file` placing arbitrary PHP on every customer site, with the operator's cookie attached. The "authenticated user" abstraction collapses without a CSRF check.
**Suggested fix**: Either (a) flip the session cookie to `sameSite: "strict"` (acceptable here — there's no SSO / external login flow that'd break), OR (b) add a double-submit CSRF token: issue one on session create, expose via `GET /api/csrf`, require `X-CSRF-Token` header on every non-GET route, validate in middleware. Strict-same-site is the smaller diff; pair it with an `Origin`-header check in middleware for belt-and-suspenders.

### [P1] `customModel` flows into `spawn(..., { shell: true })` on Windows
**File**: lib/llm-cli.ts:71-76, 102-109; lib/website-generator.ts:556 (passes operator-supplied `customModel` through to `runLlmCli`)
**Issue**: `runLlmCli` passes `model` as an argv element to `spawn("codex", ["exec", "--skip-git-repo-check", "--model", model, "-"], { shell: process.platform === "win32" })`. On Windows `shell: true` means Node serializes argv into a single command string interpreted by `cmd.exe` — and Node's argv quoting on Windows under shell mode is famously fragile. `customModel` reaches this path from `POST /api/ai-generator/queue` (`body.custom_model`), `POST /api/domains/[domain]/run-pipeline` (form/json `custom_model`), and `POST /api/domains/[domain]/run-from/[step]` — all only validated by `trimOrNull` (no shape check).
**Why it matters**: An authed operator (or anyone with a CSRF foothold per the previous P1) can submit `custom_model="gpt-5 & calc.exe"` or similar and get arbitrary command execution on the dashboard host. The host has DO API token, SA API key, Spaceship API secret, CF Global Keys, server root password — RCE here is fleet-wide compromise. Even the Linux path (`shell: false` on POSIX) only avoids it because the platform branch doesn't enable shell; deploying to Windows or any future container that flips this flag is one config change away from RCE.
**Suggested fix**: Validate `customModel` against a strict allowlist regex (`/^[A-Za-z0-9._\/@:-]{1,128}$/`) at the route layer, AND drop `shell: true` — call the `.cmd` shim directly via `spawn(binaryPath, args, { shell: false })` after resolving with `where`/`which`. The `findBinary()` helper already exists; use its return value as the spawn target.

### [P1] CSV import doesn't validate domain shape — bad strings reach SSH paths
**File**: app/api/domains/import/route.ts:64-66 + lib/repos/domains.ts:47-49
**Issue**: `addDomain(domain)` is called with any non-empty trimmed string from the CSV's `domain` column; no shape validation. Compare with `app/api/ai-generator/queue/route.ts:21` which DOES enforce `DOMAIN_SHAPE = /^(?=.{1,253}$)(?:[a-z0-9](?:[a-z0-9-]{0,61}[a-z0-9])?\.)+[a-z]{2,}$/i`. Once a malformed row lands, the value flows to `appNameFor(domain).replace(/\./g,"-").replace(/_/g,"-")` (which only strips `.`, `_`) and into shell strings: `find /home /var/www -maxdepth 6 -type d -name public_html \( -path '*${appName}*' -o -path '*${domain}*' \)` (lib/serveravatar.ts:1499-1501). A domain containing `'` breaks out of the single-quoted shell context.
**Why it matters**: Authed operator (or CSRF) with file-upload to `/api/domains/import` can plant `evil'$(id)#.com` (or use the `override-field` route which has no shape check on the URL `[domain]` either) and trigger command execution next time the pipeline runs SSH steps for that domain. Payload is durable — sits in the DB until the pipeline picks it up.
**Suggested fix**: Reuse the `DOMAIN_SHAPE` regex from `ai-generator/queue/route.ts` in `import/route.ts` (skip + count rejected rows in the response). Add the same check in `lib/repos/domains.ts:addDomain` so every entry-point enforces it. Quote `${appName}` / `${domain}` properly in the SSH shell strings — the `find ... -path` uses single quotes but a domain containing `'` defeats them; switch to a quoted-via-JSON-stringify approach or call `find` with `-name` against the literal value (no shell quoting).

### [P1] Login route is a fingerprint oracle for "no password configured"
**File**: app/api/auth/login/route.ts:30-36 + app/login/page.tsx:44-57
**Issue**: When no dashboard password is set, the login API returns `HTTP 500 {"error": "No dashboard password configured. Set one via Flask Settings page first."}`. The login page actively probes this with a sentinel password (`"__probe__no_match__"`) on mount to detect open-access mode. ALSO worth flagging: when no password is configured, the **middleware still enforces auth** (good — `/api/...` returns 401), so the dashboard isn't actually open. The "Skip to dashboard" link in the open-access UI just bounces back to /login. So the bug is informational, not access-bypass — but it's still a free signal.
**Why it matters**: An unauthenticated attacker scanning the public IP can fingerprint deploy state in a single POST: 500 → "fresh deploy, race the operator to set a password," 401 → "password set, brute-force." Combined with the 5/60s login throttle (and the throttle bucket keyed on `x-forwarded-for` which is spoofable per [P2] below), a slow distributed brute-force is feasible.
**Suggested fix**: Return the same generic 401 ("Invalid password") whether the password row is missing or wrong. Move the probe-by-sentinel UX to a session-authed `/api/settings/security-status` instead. While you're there, log the unconfigured-password case to the audit log on every probe so an admin notices someone scanning.

## P2 — Should fix

### [P2] `x-forwarded-for` taken raw — login throttle bypass + audit log forgery
**File**: app/api/auth/login/route.ts:10, every other route that grabs `req.headers.get("x-forwarded-for")?.split(",")[0]?.trim()`
**Issue**: Throttle bucket key + audit `actor_ip` come from the first comma-separated segment of `X-Forwarded-For` with no proxy whitelist. If the dashboard is exposed without a trusted reverse proxy stripping forged values (or if it's behind a proxy that *appends* rather than replacing), an attacker can rotate `XFF: 1.2.3.4`, `XFF: 1.2.3.5`, ... and get a fresh 5-attempts-per-minute bucket per spoofed IP — effectively unbounded brute force.
**Why it matters**: Defeats the only rate-limit on the password endpoint. Audit log becomes untrustworthy for incident response (every entry says whatever the attacker wanted).
**Suggested fix**: Read `XFF` only when behind a configured trusted proxy (gate via `SSR_TRUST_PROXY=1` env var); otherwise fall back to `req.ip` / connection-level address. Document the deployment requirement (must terminate TLS at a proxy that strips client-supplied XFF) in DEPLOY.md. Consider using `loginThrottle` keyed by both IP and a per-process global counter so a fully-spoofed flood still hits a ceiling.

### [P2] No security headers (CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy)
**File**: next.config.mjs (entire file), middleware.ts (no header injection)
**Issue**: `next.config.mjs` only configures TS strictness, image optimization, and serverExternalPackages — no `headers()` block. Middleware doesn't add security headers either. `next dev` and `next start` ship default headers only.
**Why it matters**: Missing CSP makes any future XSS catastrophically easier to weaponize (the dashboard renders LLM-generated previews and audit-log strings — both attacker-influenced). Missing X-Frame-Options allows clickjacking the dashboard inside a malicious iframe (combined with `sameSite: lax` cookie, an attacker can frame-jack a "click here for free credit" button into firing `/api/servers/destroy-all`). No HSTS means a downgrade attacker on the operator's network can MITM the login.
**Suggested fix**: Add a `headers()` export to next.config.mjs returning `Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' (or hashed); connect-src 'self' https://api.cloudflare.com https://api.anthropic.com ...; frame-ancestors 'none'`, `Strict-Transport-Security: max-age=31536000; includeSubDomains`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`. Apply globally with `source: "/(.*)"`. Tune `connect-src` from the actual external endpoints listed in lib/website-generator.ts and lib/serveravatar.ts.

### [P2] `server_ip` not validated as a known dashboard server (SSH/SFTP to arbitrary IPs)
**File**: app/api/sa/index-file/route.ts:14-55, app/api/sa/upload-file/route.ts:25-87, app/api/sa/service-restart/route.ts:15-50, app/api/sa/bulk-edit/route.ts:20-79
**Issue**: All these routes accept a free-form `server_ip` string and pass it to `openSsh(ip)` (lib/sa-control.ts:83-89), which connects with `username: "root"` + the global `server_root_password` setting. No check that the IP is in `servers.ip` for the dashboard's known fleet.
**Why it matters**: An authed operator (or CSRF foothold) can target ANY IP — including internal RFC1918 addresses or cloud-metadata-like ranges — with the dashboard's stored root password. If that password is reused across non-dashboard hosts (likely, since it's a single global setting), this enables lateral movement. Even without password reuse, repeated failed root logins from the dashboard's IP can trip fail2ban / abuse alarms on third parties, getting the dashboard host blocklisted.
**Suggested fix**: At the route layer, look up `server_ip` in the `servers` table (`SELECT 1 FROM servers WHERE ip = ?`) and reject 403 if not present. Same check belongs in `lib/sa-control.ts:openSsh` as a defense-in-depth gate so future callers can't bypass.

### [P2] Open redirect in login `?next=`
**File**: app/login/page.tsx:34, 75
**Issue**: `next = params.get("next") || "/"`, then `window.location.assign(next)` after a successful login. No same-origin check. The middleware sets `next` to a path-only value when bouncing unauthed users (`loginUrl.searchParams.set("next", pathname)`), so the legitimate flow is path-only — but the value is fully attacker-controllable from the URL.
**Why it matters**: `https://dashboard.example.com/login?next=https://evil.com/login-clone` — operator types password (which works), gets redirected to evil.com, lands on a clone, types password again. Phishing assist. Also classic for credential theft chains paired with a malicious browser extension scanning "next" params.
**Suggested fix**: Validate `next` is a same-origin path: `if (!next.startsWith("/") || next.startsWith("//")) next = "/"`. Apply before `window.location.assign`.

### [P2] Server-create `name` taken raw, no shape validation
**File**: app/api/servers/create/route.ts:19, 38
**Issue**: When the operator supplies an `explicitName`, it's `.trim()`'d and passed straight to the job payload (and downstream to DO's droplet API and SA's server-name field). No length cap, no character allowlist. DO and SA each have their own validation but error messages bubble back through.
**Why it matters**: Lower than the SSH-shell case (no direct command execution), but DO droplet names show up in SSH banner / hostname / cloud-init scripts in some configurations. A malicious name could embed log-poisoning sequences (ANSI escapes in operator terminals tailing logs) or shell-injection-shaped strings that a future automation script naively interpolates.
**Suggested fix**: Match the auto-generator's shape — `/^[a-z0-9-]{1,63}$/`. Reject otherwise.

## P3 — Nits

### [P3] Settings POST has no audit of WHICH fields changed (only count)
**File**: app/api/settings/route.ts:132 (`appendAudit("settings_save", "", \`${count} fields submitted\`, ip)`)
**Issue**: Audit log records `count=N` without listing field names. Forensics after a credential rotation incident has no way to tell what was rotated when.
**Suggested fix**: Log the field names list (NOT values). `appendAudit("settings_save", "", \`fields=[${changedKeys.join(",")}]\`, ip)`.

### [P3] Boot self-bootstraps the Fernet keyfile if missing
**File**: lib/secrets-vault.ts:97-109
**Issue**: First-boot generates a fresh Fernet key and writes it to `data/.ssr_secret_fernet`. Convenient for dev; in prod this means a wiped data dir silently regenerates a NEW key, after which every encrypted column decrypts to "" (the catch in `decrypt()`). Forgotten backups → silent credential loss without an obvious "this is a fresh key, your secrets are gone" warning.
**Suggested fix**: In production (NODE_ENV=production), refuse to auto-generate if the keyfile is missing AND any rows in `settings` already have the `enc:v1:` marker — log a clear "key missing but encrypted rows present" error and exit. Auto-gen is fine for fresh installs.

## Clean lenses
- **SQL injection**: parameterized queries throughout (better-sqlite3-style `?` placeholders); the only string-built clause is `searchAuditLog`'s WHERE construction in lib/repos/audit.ts:19-31 and it appends `?` placeholders, never user values.
- **Path traversal in website-generator's `validateGeneratedFiles`**: the `SAFE_PATH = /^[a-zA-Z0-9._\-/]+$/` allowlist + the explicit `path.includes("..")`, `path.startsWith("/")`, `path.endsWith("/")`, `segments.length > 6`, `segments.some(s => s.length === 0)` checks (lib/website-generator.ts:430-447) are airtight against `..`, absolute paths, empty segments, and duplicates. Unicode/double-encoding would never satisfy the ASCII-only allowlist.
- **SSRF in CF / SA / DO calls**: every external `fetch()` uses a hardcoded host (`api.cloudflare.com`, `api.anthropic.com`, `api.openai.com`, `openrouter.ai`, `api.moonshot.ai`, `generativelanguage.googleapis.com`); only the path / account ID is interpolated, and the path can't escape the host. `cf-keys/[id]/refresh-status/route.ts:66` does `fetch(\`https://${d.domain}/\`)` but `d.domain` comes from a `WHERE cf_key_id = ?` lookup, not request input.
- **Auth gating completeness**: middleware.ts catches everything except a tight allowlist (`/login`, `/api/auth/login`, `/api/health`, `/healthz`, `/favicon.ico`) and Next static asset prefixes. The matcher explicitly handles `/api/:path*` to avoid the dot-in-path Next quirk. Every state-changing route I sampled relies on the middleware alone — consistent surface, no per-route ad-hoc checks to drift.
- **Multi-tenancy**: confirmed single-operator by design (login-page copy: "Single-operator tool. No SSO, no signup, no recovery.") — no per-user ownership model expected, no IDOR class to flag.
- **CORS**: no `Access-Control-Allow-Origin` set anywhere; defaults are same-origin only.
- **Cookie hardening basics**: `httpOnly: true`, `secure: process.env.NODE_ENV === "production"`, 8-hour `maxAge`, lazy-resolved `SSR_SESSION_SECRET` with prod-only enforcement of ≥32 chars (auth-config.ts:23-32). Only sameSite is downgraded from ideal (`lax` instead of `strict`) — covered in the CSRF P1.
- **Password storage**: PBKDF2-SHA256 (600k iters) Werkzeug-format hash with `crypto.timingSafeEqual` comparison; scrypt fallback also constant-time. Login throttle is atomic via `loginThrottleCheckAndReserve` (closes the parallel-request race that a naive check-then-reserve would have).
